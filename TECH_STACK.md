# 技术栈与工程规范（NMTI）

本文档记录本项目约定的技术选型、本地开发、部署与发布流程，便于团队对齐与后续演进。

---

## 技术栈（通用轻量前端）

| 层级 | 选型 |
|------|------|
| 前端基础 | HTML5 + CSS3 + JavaScript（ES6 Modules） |
| 渲染层 | Canvas API（适合游戏、可视化、交互原型） |
| 音频层 | Howler.js（可选，统一音效管理） |
| 架构方式 | 事件驱动（EventBus）+ 状态管理（简单状态机） |
| 配置管理 | 集中配置文件（如 `config.js`） |
| 包管理 / 构建 | 可先无构建（纯静态），后续按需接入 Vite / Webpack |
| 版本控制 | Git + GitHub |

---

## 本地开发方案（通用）

1. 启动静态服务器（任选其一）：
   - `python -m http.server 8080`
   - `npx serve .`
2. 浏览器访问本地地址进行联调。

### 建议保留的基础检查

- 控制台无报错
- 关键交互可跑通
- 响应式基础检查（桌面 / 移动）

---

## 部署方案（通用 GitHub Pages）

| 项 | 说明 |
|----|------|
| 托管方式 | 静态站点部署到 GitHub Pages |
| 触发策略 | `main` 分支 push 自动部署（也支持手动触发） |
| CI/CD 工具 | GitHub Actions |

### 典型流程

1. Checkout 代码  
2. （可选）执行构建  
3. 上传静态产物  
4. 部署到 Pages  

### 版本可追溯（建议）

- 部署时生成 `version.json`（含 commit hash / build time）
- 页面显示版本标识，便于回滚和定位问题

---

## 可直接复用的发布规范（建议）

- 默认在开发分支开发，检查通过后合并 `main`
- 仅 `main` push 触发正式发布

### 每次发布要求

- 有对应 commit
- 有最小回归结果
- 有变更记录（Release Notes）
